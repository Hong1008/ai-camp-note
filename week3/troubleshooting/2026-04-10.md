# [Senior Mentor] 2026-04-10 트러블슈팅 리포트

오늘의 실습 과정에서 발생한 주요 기술적 이슈와 해결 과정을 정리했습니다.

---

## 1. Pydantic-Pandera 메타클래스 충돌 (Metaclass Conflict)
- **현상**: `class Schema(pa.DataFrameModel, BaseModel)` 처럼 두 라이브러리를 상속받으려 할 때 `metaclass conflict` 에러 발생. 또한 `from_model` 속성이 Pandera 버전에 따라 잘못된 체크 항목으로 인식되어 `AttributeError` 발생.
- **원인**: Pydantic과 Pandera 모두 복잡한 메타클래스를 사용하여 동적으로 클래스를 빌드하기 때문에 직접적인 다중 상속이 불가능함.
- **해결**: Pydantic DTO 레이어를 과감히 제거하고 전수 데이터를 `List[dict]` 형태로 받은 뒤, Pandera `DataFrameModel` 단일 스택으로 검증 및 데이터프레임 변환을 통합하여 단순화함.
- **교훈**: 중첩된 데이터 수집 시 불필요한 직렬화(Serialization)는 성능과 안정성 모두를 해칠 수 있음. 단일 소스 오브 트루스(SSOT)를 유지하는 것이 아키텍처 상 유리함.

## 2. API 문자열 숫자의 타입 불일치 (SchemaError)
- **현상**: 서울시 API나 공공데이터 API에서 숫자 값을 `"123"`과 같은 문자열로 반환하여 Pandera 타입 검증(`int`)에서 에러 발생.
- **원인**: 공공기관 API의 원본 데이터 타입이 엄격하지 않아 JSON/XML 응답 시 모두 문자열로 처리되는 경우가 많음.
- **해결**: Pandera Schema의 `Config` 클래스에 `coerce = True` 옵션을 추가하여 자동으로 형 변환을 수행하도록 설정함.
- **교훈**: 외부 API 데이터를 다룰 때는 항상 데이터 타입의 불확실성을 염두에 두고 '강제 변환(Coercion)' 전략을 기본으로 사용해야 함.

## 3. 환경 변수 명령 불일치 (Config Mapping)
- **현상**: `.env` 파일의 오타(`NAVAR_CLINET_ID`) 및 클라이언트별 개별 호출로 인한 관리 복잡성 발생.
- **원인**: 프로젝트 초기 설정 시 오타 발생 및 설정 관리의 파편화.
- **해결**: `config.py`를 중앙 설정 저장소로 구축하고, 오타가 난 환경 변수를 올바른 내부 변수 명칭(`NAVER_CLIENT_ID`)으로 매핑하여 코드 전반의 가독성을 높임.
- **교훈**: `os.getenv`는 한곳에서만 수행하고, 나머지 코드는 중앙 관리 객체를 통해서만 설정값을 가져오는 것이 유지보수에 유리함.
