## 1. Jupyter 환경에서의 Playwright Sync API 충돌
- **현상**: Jupyter 셀에서 `sync_playwright().start()` 실행 시 `RuntimeError: This event loop is already running` 발생.
- **원인**: Jupyter(IPython)는 자체적으로 `asyncio` 이벤트 루프 위에서 돌아감. Playwright의 Sync API는 내부적으로 새로운 루프를 중첩 실행하려 하므로 충돌이 발생함.
- **해결**: Jupyter 환경에서는 기본적으로 **Async API**를 사용하며, Jupyter의 **Top-level await** 지원 기능을 활용하여 `await` 키워드로 즉시 호출하는 방식을 적용함.
- **교훈**: 실행 환경(REPL vs 스크립트)에 따라 이벤트 루프의 점유 상태가 다를 수 있음을 이해하고 적절한 인터페이스(Sync vs Async)를 선택해야 함.

## 2. 동적 데이터(좋아요 수) 수집 전략 부재
- **현상**: 멜론 차트에서 BeautifulSoup 정적 파싱만으로는 '좋아요' 수가 모두 `0`으로 수집되거나 데이터 누락 발생.
- **원인**: 좋아요 데이터는 초기 HTML에 포함되지 않고, 페이지 로드 후 별도의 JSON API를 통해 비동기적으로 채워지는 구조임.
- **해결**: 브라우저 네트워크 로그(XHR) 분석을 통해 좋아요 API 엔드포인트를 찾아내고, **Batch Request 기법**(100개의 ID를 콤마로 묶어 송신)을 사용하여 단 한 번의 호출로 모든 데이터를 수집함.
- **교훈**: 화면에 보이는 모든 데이터가 HTML 소스에 있는 것은 아님. 네트워크 패킷 분석을 통해 서버와 클라이언트 간의 실제 데이터 흐름을 파악하는 것이 스크래핑의 핵심임.

## 3. 빌보드 차트의 유동적 DOM 구조 대응
- **현상**: 기존의 `h3#title-of-a-story`와 같은 ID 기반 선택자가 작동하지 않아 빈 데이터프레임이 반환됨.
- **원인**: 빌보드와 같은 대형 글로벌 사이트는 A/B 테스트나 잦은 업데이트로 인해 ID나 복잡한 유틸리티 클래스명이 자주 변경됨.
- **해결**: 클래스명의 완전 일치보다는 **상대적 계층 구조**를 활용함. 고유한 스타일을 가진 `h3` 요소를 먼저 찾고, `find_next_sibling('span')` 등을 사용하여 주변 데이터를 탐색하는 '견고한(Robust) 선택자' 전략을 적용함.
- **교훈**: 변화에 민감한 선택자(Brittle Selector)보다는 구조적 선후 관계를 활용한 선택자가 코드의 생명주기를 더 길게 보장함.

## 4. Python Bare `except:` 사용 지양 (PEP 8)
- **현상**: `except: continue`와 같이 예외의 종류를 명시하지 않은 소위 'Bare except' 구문 사용.
- **원인**: 모든 예외를 한꺼번에 무시하여 코드 작성을 단순화하려는 시도.
- **해결**: 시스템 종료 신호(`SystemExit`, `KeyboardInterrupt`)를 가로채지 않도록 **`except Exception:`**으로 수정하고, 필요한 경우 로그를 남기도록 개선함.
- **교훈**: 예외 처리는 '모르는 에러를 숨기는 것'이 아니라, '예상 가능한 에러를 관리하는 것'임. 명확한 예외 명시는 버그 추적의 첫걸음임.

## 5. 봇 탐지 우회 및 헤더 신뢰성 확보
- **현상**: `requests.get()` 호출 시 데이터 대신 403 Forbidden 에러 혹은 차단용 캡차 페이지 노출.
- **원인**: 서버 측에서 `User-Agent`가 파이썬 라이브러리인 것을 감지하고 비정상 접근으로 간주하여 차단함.
- **해결**: 실제 최신 브라우저에서 추출한 `User-Agent` 값을 `headers` 딕셔너리에 담아 요청마다 명시적으로 전달함.
- **교훈**: 웹 스크래핑 시 서버에 부담을 주지 않는 배려(매너)와 함께, 정상적인 사용자 접근처럼 보이게 하는 세심한 설정이 필수적임.
